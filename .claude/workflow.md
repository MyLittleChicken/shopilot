# 개발 방식

스펙 우선(SDD)과 테스트 우선(TDD)을 혼합한다. 스펙을 먼저 정의하고, 그 스펙으로 테스트를 쓴 뒤, 테스트를 통과시키는 구현을 한다. **스펙·테스트 없이 구현부터 시작하지 않는다.**

## 기능 단위 루프

각 기능은 `main`에서 작업 브랜치를 분리해 진행하고, 구현·테스트가 끝나면 PR로 합친다. `main`에 직접 커밋하지 않는다.

0. **브랜치 분리** — `main`에서 작업 브랜치를 딴다 (예: `feat/<기능>`).
1. **스펙 (SDD)** — `docs/specs/<기능>.md`에 먼저 적는다.
   - 무엇을·왜, 입출력과 계약, 수용 기준(acceptance criteria), 비범위(out of scope).
   - 계약의 실체는 `packages/schemas`의 Zod 스키마다. 스펙은 그것을 사람이 읽는 설명이다.
2. **태스크 분해** — 스펙 안의 수용 기준을 작은 태스크로 쪼개 그 스펙 파일에 체크리스트로 둔다(기능별로 바운드되어 비대해지지 않는다). `docs/PROGRESS.md`에는 현재 집중과 스펙 인덱스만 가볍게 유지한다.
3. **테스트 (red)** — 각 수용 기준을 실패하는 테스트로 먼저 쓴다. 원칙은 `.claude/testing.md`.
4. **구현 (green)** — 테스트를 통과시키는 최소 구현.
5. **리팩터** — 테스트 통과를 유지하며 정리한다.
6. **PR** — 구현과 테스트(green)가 끝나면 PR을 연다. 리뷰·CI 통과 후 `main`에 머지한다.

## 원칙

- 스펙 → 테스트 → 구현 순서를 지킨다.
- 한 번에 하나의 작은 단위. 테스트가 초록일 때 다음으로 넘어간다.
- 비결정론적 동작(LLM 등)은 결과 내용이 아니라 계약·구조로 검증한다.
- 커밋도 같은 순서로 의미 단위로 나눈다(스펙 → 테스트 → 구현).
- 작업은 브랜치에서 하고, `main` 병합은 PR로만 한다 (직접 커밋 금지).

## 산출물 위치

- 스펙: `docs/specs/<기능>.md`
- 계약: `packages/schemas`
- 세부 태스크: 해당 스펙 파일의 수용 기준 체크리스트
- 진행 대시보드: `docs/PROGRESS.md` (현재 집중 · 마일스톤 · 스펙 인덱스)
- 시간순 이력: git 히스토리
- 테스트: 대상 가까이 `*.test.ts`
